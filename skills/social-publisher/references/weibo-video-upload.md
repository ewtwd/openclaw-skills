# 微博视频上传专项参考

## 目标

在 OpenClaw `browser profile=user` 模式下，尽量自动化完成微博视频上传。

## 已验证现象

1. 微博页面常存在隐藏的 `input[type=file]`
2. 该 input 的 `accept` 可能同时包含图片和视频，例如：
   - `image/*`
   - `video/mp4`
   - `video/*`
   - `.mkv`, `.flv`
3. 点击可见“视频”文字入口时，不一定会触发一个可被 `browser.upload` 直接接管的文件选择器
4. 即使文件成功注入隐藏 input，也**不等于**页面已经真正进入“视频上传队列”

## 推荐攻击顺序

### 1. 先走常规上传

- 打开微博发布区
- 点击“视频”入口
- 如果页面暴露出可上传的 inputRef，先直接 `browser.upload`

### 2. 查隐藏 file input

用 evaluate 查：
- `document.querySelectorAll('input[type=file]')`
- 记录：
  - `accept`
  - `multiple`
  - class / outerHTML 片段

如果 accept 含视频类型，优先把它当作视频上传原始入口。

### 3. 代理 input 方案

如果 existing-session 模式下拿不到原始 input 的 ref，或原始控件无法直接吃 `browser.upload`：

- 在页面里注入一个临时可见 `input[type=file]`
- 赋予明确 aria-label，例如：`OpenClaw视频上传`
- 对代理 input 执行 `browser.upload`
- 在代理 input 的 `change` 事件里：
  - 建立 `DataTransfer()`
  - 把 `proxy.files` 转交给 `original.files`
  - 对原始 input 触发：
    - `input`
    - `change`

## 成功判定

只有满足以下条件组合，才能认为“视频已挂上并可发送”：

- 出现视频相关信号，例如：
  - 视频缩略图
  - 时长 / 文件名 / 封面预览
  - 视频编辑区域
- 同时**不能**仍处于上传中 / 处理中状态，例如：
  - `上传中`
  - `视频处理中`
  - `处理中`
  - 进度条仍存在
- 同时发送按钮处于可点击状态

注意：
- 仅出现 `video` 元素还不够
- 视频已被页面接管 ≠ 视频已经上传完成

## DOM / UI 成功信号采集清单

每次怀疑视频上传成功后，都优先做下面几类检查：

### 1. 文案区附近是否出现新块

重点看发布框周围是否新增：
- 缩略图容器
- 封面容器
- 时长文本，例如 `00:15`
- 上传进度文本，例如 `%`、`上传中`
- 删除视频 / 重传按钮
- 封面编辑按钮

### 2. 页面里的可见文本变化

可搜索这些关键词：
- `上传中`
- `上传完成`
- `视频处理中`
- `封面`
- `重新上传`
- `删除`
- `时长`
- `预览`

### 3. DOM 结构变化

上传前后对比：
- `input[type=file]` 的数量是否变化
- 发布区下是否新增 `video`、`img`、`canvas`、进度条节点
- 发布按钮是否从不可用变为可用
- 页面是否新增弹层 / 编辑区 / 裁剪区

### 4. 网络/状态联动（如后续工具支持）

优先关注：
- 媒体上传请求是否发出
- 是否返回 media id / object id
- 上传后是否进入“封面/描述”阶段

## 不能算成功的情况

以下情况都不能对老板说“上传成功”：

- 只是 `browser.upload` 返回 `ok: true`
- 只是代理 input 接受了文件
- 只是原始 input 收到了 `change`
- 只是页面没报错
- 只是发布页/tab 被打开
- 只是点击了“视频”入口

这些最多只能说：
- “文件已注入页面，待确认微博是否接收”
- “视频发布入口已打开，但还没看到微博接管文件的证据”

## 失败后的标准回复

如果多轮尝试后仍无上传信号：

- “视频入口已尝试，文件也已注入页面，但微博没有进入可见上传状态。”
- “当前不能确认上传成功，我不建议直接点发送。”
- “你手动选一下视频，我继续负责后面的发送。”

## 本次实战已确认的现象（2026-03-25）

- 用户明确看到“视频发布 tab 页面已经弹出”，但自动化侧**仍未看到可确认的视频上传成功信号**
- 通过 evaluate 能查到隐藏的 `input[type=file]`，且其 `accept` 同时包含图片和视频类型
- 通过代理 input + `DataTransfer` + 向原始 input 触发 `input/change`，可以做到“文件注入页面”
- 但注入后页面**没有立即出现可见的缩略图、进度条、时长或视频编辑区**
- 结论：目前最多只能判定为“文件可能已交给前端”，**不能判定微博已经真正接管并进入上传流程**

## 后续优化方向

1. 识别微博视频上传成功后的 DOM 特征
2. 区分图文 input 与视频 input 是否为同一个控件
3. 捕获上传后的网络请求特征（如可行）
4. 找到视频发布页中真正的状态区 / 封面区 / 发布按钮联动条件
5. 如有必要，单独做一个微博视频发布专项脚本/流程
